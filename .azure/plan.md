## 4. Authentication & Authorization Design

### 4.1 Entra External ID App Registration

| Property | Value |
|---|---|
| App name | Allowance App |
| Tenant | bleytech.onmicrosoft.com |
| Supported account types | Microsoft accounts only |
| Platform | Single-page application (SPA) |
| Redirect URIs | `https://<swa-hostname>/auth/callback`, `http://localhost:5173/auth/callback` |
| Front-channel logout URL | `https://<swa-hostname>/auth/logout` |
| API scopes exposed | `api://<client-id>/AllowanceApp.Access` |
| ID Token | Enabled |
| Access Token | Enabled |

### 4.2 MSAL Configuration (Frontend)

- Authority: `https://bleytech.ciamlogin.com/`
- Flow: Authorization Code + PKCE (public client, no client secret)
- Scopes requested: `openid`, `profile`, `offline_access`, `api://<client-id>/AllowanceApp.Access`
- Token storage: `sessionStorage` (more secure than localStorage for SPA)
- Silent token refresh via refresh token rotation

### 4.3 Backend JWT Validation

Every HTTP trigger function validates:
- `iss` — must match External ID issuer (`https://bleytech.ciamlogin.com/<tenant-id>/v2.0`)
- `aud` — must match app client ID
- `exp` / `nbf` — current time within window
- `oid` — used to look up user's `familyId` and `role` from Cosmos DB `users` container
- Signature — validated against External ID JWKS endpoint (cached, refreshed on key rotation)

### 4.4 Role Enforcement

- `FamilyAdmin` required for: `addTransaction`, `editTransaction`, `deleteTransaction`, `updateSettings`, `getAuditLog`
- `User` (or `FamilyAdmin`) for: `getFamily`, `getTransactions`
- `allowanceScheduler` has no inbound auth (timer trigger, runs server-side)

### 4.5 Family Data Isolation

- Every API call resolves `oid` → `{ familyId, role }` via the `users` Cosmos DB container
- All Cosmos DB queries are scoped with a `familyId` filter — no cross-family data access possible
- Partition key for all containers is `/familyId`

## 6. Allowance Scheduler

- **Trigger**: Cron `*/5 * * * * *` (every 5 minutes)
- **Logic**:
  1. Query `users` container for all users where `kidSettings.nextAllowanceDate <= now` and `kidSettings.allowanceEnabled = true`
  2. For each: create a `transactions` document (Income, amount, notes = "Automatic allowance")
  3. Compute next `nextAllowanceDate` based on frequency + day/time/timezone settings
  4. Update user's `kidSettings.nextAllowanceDate`
- **Idempotency**: Check that no transaction with same `kidOid` + `date` (±1 min window) exists before inserting
- **No Azure portal changes needed** to modify schedules — all config is in the database

## 9. Security Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Auth flow | Auth Code + PKCE | Secure for SPA; no client secret needed |
| Token storage | sessionStorage | Avoids persistent token in localStorage; reduced XSS risk |
| Role source | Cosmos DB (not token claims) | Token claims require admin consent setup; DB is more flexible |
| Secrets | Key Vault references in Function App settings | Never hardcoded; managed identity access |
| Data isolation | Server-side familyId scoping from `oid` | Client cannot manipulate which family data it sees |
| JWKS caching | In-memory per Function instance, refresh on 401 | Resilient to key rotation, avoids per-request JWKS fetch |
| CORS | Static Web Apps proxy (`/api/`) | No cross-origin requests; Functions not directly exposed |
